Implementation Notes

2026-05-23: @GraphStored didSet support

Design Decisions

  • @GraphStored already expands into explicit get/set accessors backed by the generated $property node. Supporting didSet is therefore being interpreted as preserving the user's observer body and running it from the generated setter after assigning newValue.
  • willSet support is being implemented in the same generated setter. The observer body runs before writing to $property.wrappedValue, with Swift's default newValue binding or the custom willSet(customName) binding.
  • When both observers are present, generated code should run willSet, capture oldValue, assign the new value, then run didSet.
  • The generated setter should capture the old value before assignment and expose it as Swift's normal oldValue name, or as the custom observer parameter name when the source uses didSet(customName).
  • Initialization accessors remain observer-free, matching Swift's stored-property behavior where didSet is not called during initialization.
  • Macro expansion coverage is kept at the accessor-expansion level by calling UnifiedStoredMacro.expansion(...providingAccessorsOf:) directly. This verifies generated get/set accessors without snapshotting an invalid-looking full property body.

Deviations

  • No intentional deviations for willSet are planned. External storage notifications from UserDefaults still follow the existing storage update path and do not synthesize property observer calls.

Trade-offs

  • Using the generated setter keeps behavior local to macro expansion and avoids adding a new runtime hook just for property observer syntax.
  • The lower-level Stored.onDidSet remains available, but macro-level didSet gives source compatibility with normal Swift property observers.
  • Swift accessor macro expansion keeps the original didSet observer syntax visible in macro snapshots. The implementation does not rely on that observer being invoked; it explicitly runs the observer body from the generated setter so runtime behavior is covered by tests.
  • Because handwritten Swift does not allow didSet and get to coexist in the same accessor block, macro snapshot tests should not assert that invalid-looking expanded form. The meaningful coverage is compile/runtime behavior for memory and UserDefaults backed properties.

Open Questions

  • Confirm whether property observers should ever run for external storage updates, such as UserDefaults changes made outside the generated setter.
